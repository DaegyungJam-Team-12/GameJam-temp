# 기술 구조와 협업 잠금

> **상태: LOCKED · P0 v1.0 · 2026-07-18**  
> 3명이 같은 씬과 프리팹을 건드리지 않는 것을 최우선으로 한다. 새 프레임워크나 범용 아키텍처는 만들지 않는다.

## 1. 기술 스택

- Unity 6 + C#
- 창 제어: UniWindowController
- 데이터: 로컬 JSON
- 버전 관리: GitHub 기능 브랜치·개별 worktree·일 1회 통합
- P0 플랫폼: Windows 10/11

새 패키지, DI 프레임워크, 온라인 백엔드, 분석 SDK는 P0에 추가하지 않는다.

## 2. 런타임 모듈

| 모듈 | 책임 | 소유자 |
|---|---|---|
| `IceFieldController` | 최대 20개 스폰·재스폰·티어 혼합·풀링 | 개발자 1 |
| `IceActor` | 클릭·홀드 피해, 균열, 파괴, 특수빙 반응 | 개발자 1 |
| `IceEffectResolver` | 직접·보조·연쇄 효과, 재귀 3단 제한 | 개발자 1 |
| `GameStateController` | 7단계 흐름, 마지막 5초 이벤트, 60초 작업, 표준 30초 항해 | 개발자 2 |
| `RouteProgressService` | 전진 점수, 고정 항로 3개, 도착·완료 플래그 | 개발자 2 |
| `UpgradeService` | 정비 자금, 구매 13개, 조건·효과 값 | 개발자 2 |
| `SaveService` | 로컬 저장, 버전, 오프라인 대기 시간 | 개발자 2 |
| `WindowController` | 도킹·확장·DPI·멀티모니터·폴백 | 개발자 3 |
| `UIStatePresenter` | 상태별 HUD·지도·정비·결과 표시 | 개발자 3 |
| `AudioController` | 음소거 기본값, SFX·앰비언스 재생 | 개발자 3 |

모듈은 필요 이상의 인터페이스 계층을 만들지 않는다. 데이터 모델과 이벤트 계약만 공유하고 각 담당자가 자기 영역 내부 구현을 소유한다.

## 3. D1 공유 계약

개발자 2가 초안을 만들고 세 개발자가 D1 안에 확정한다.

- `RunResult`: 운항 보수, 파괴 수, 최고 빙질, 최대 연쇄, 전진 점수, T3 첫 파괴 여부
- `RouteState`: 현재 항로 ID, 진행값, 목표값, 도착 상태
- `BalanceData`: 타이머, 빙질, 스폰, 특수빙, 정비 비용, 항로 목표
- `GameEvents`: 작업 시작·얼음 파괴·작업 종료·구매·항로 도착

이벤트에는 게임 규칙을 넣지 않는다. 개발자 1은 `RunResult` 원천값을 제공하고, 개발자 2가 정산·항로·저장을 확정하며, 개발자 3은 결과를 표시한다.

## 4. 창 구현

- 도킹 480×56 / 확장 720×480
- taskbar 위 작업영역 계산, 멀티모니터와 100%·125%·150% DPI 확인
- 알파 클릭 통과는 사용하지 않음. 창 전체가 인터랙션 영역
- 도킹·항해 중 `targetFrameRate` 10~15, 쇄빙 작업 중 60fps
- OS 의존 코드는 `WindowController` 한 영역으로 격리
- 폴백: taskbar 고정 실패 시 하단 스냅 일반 창. 코어 게임은 그대로 플레이 가능
- D2까지 폴백 포함 동작하지 않으면 taskbar 특수 효과 개발을 중단하고 일반 창 폴백을 P0 기준으로 확정

## 5. 씬·프리팹·아트 통합 규칙

- **개발자 3만** 메인 Unity 씬, UI 프리팹, Canvas 계층을 수정한다.
- 개발자 1·2는 스크립트·데이터·테스트 씬 또는 독립 테스트 프리팹만 수정한다.
- 아티스트 원본은 `SourceArt`에서 관리하고 Unity 임포트·슬라이스·압축 설정은 개발자 3이 담당한다.
- 씬에서 연결이 필요한 기능은 이벤트 계약과 연결 가이드를 전달하고 개발자 3이 통합한다.
- `.meta` 파일을 수동 삭제·이름 변경하지 않는다.

## 6. 데이터와 저장

### 데이터 파일

- `balance_standard.json`: 30초 대기와 30~60분 목표의 초기 튜닝값
- `balance_demo.json`: 10초 대기·단축 항로·`ceil(표준 비용 × 0.1)` 초기값
- `routes.json`: 목적지 3개, 화물, 항로 순서
- `localization_ko.json`, `localization_en.json`: 화면 문자열

프로필은 빌드 설정 또는 실행 인자로 선택하며 공개 설정 화면에는 노출하지 않는다. 기능 분기는 만들지 않는다. 저장 파일은 `save_standard.json`과 `save_demo.json`으로 분리해 시연 진행이 표준 진행을 오염시키지 않게 한다.

### 저장 필드

- `saveVersion`
- 보유 정비 자금과 구매한 정비 항목
- 최고 해금 빙질
- 현재 항로 ID·진행값·완료 항로
- 다음 유빙 구간 준비 시각
- 최대 연쇄·총 파괴 수 등 최소 기록
- T3 첫 파괴 기록·3개 항로 완료 여부
- 선택 엔딩을 구현한 경우에만 엔딩 확인 여부
- 언어·볼륨·창 고정·화면 흔들림 설정

앱 종료 중에는 타임스탬프 차분으로 대기 시간만 진행한다. 자동 파쇄, 정비 자금, 항로 진행은 발생하지 않는다. 저장 버전이 맞지 않는 개발 빌드는 안전하게 초기화할 수 있어야 한다.

## 7. 성능 기준

- 쇄빙 작업: 목표 60fps, 1% low 45fps 이상
- 도킹·항해: 10~15fps 절전
- 화면 동시 얼음 최대 20개
- 얼음·파편·피해 숫자·공용 파티클은 오브젝트 풀 사용
- 연쇄 20개 상황에서 매 프레임 할당과 중복 물리 탐색을 피한다.
- D2 프로토타입과 D10 콘텐츠 통합 뒤 동일한 최악 조건으로 다시 측정한다.

## 8. 브랜치와 소유권

| 담당 | 브랜치 예시 | 주 수정 영역 |
|---|---|---|
| 개발자 1 | `feature/ice-combat-core` | 얼음·파쇄·특수 효과·풀링 |
| 개발자 2 | `feature/route-progression` | 상태·항로·정비·저장·데이터 |
| 개발자 3 | `feature/window-ui-integration` | 창·메인 씬·UI·아트/오디오 통합·빌드 |

- 각자 별도 worktree를 사용한다.
- D1 계약 확정 뒤 담당 외 파일을 직접 고치지 않는다. 변경이 필요하면 소유자에게 요청한다.
- 매일 한 번 개발자 3의 통합 브랜치에 순서대로 병합하고, 씬 충돌은 개발자 3이 해결한다.
- D11 콘텐츠 동결, D13 RC 동결 이후에는 차단 버그만 수정한다.

## 9. 검증

### 자동 검증

- 7단계 상태 전이, 마지막 5초 이벤트, 60초·30초 타이머
- 항로 점수 반영, 초과 점수 미이월, 북쪽 기지 목표값 완료 조건
- 구매 조건·비용·효과 값
- 저장/불러오기·버전·오프라인 대기 시간
- 표준·시연 JSON 역직렬화와 필수 필드

### 수동 검증

- 클릭·홀드·연쇄 파괴 손맛과 20개 최악 성능
- 도킹·확장·포커스·멀티모니터·DPI
- 한국어·영어 텍스트 넘침
- 소리 기본 꺼짐과 설정 유지
- 신규 저장으로 표준 30~60분 완주
- 시연 프로필로 10분 안에 전체 흐름 리허설
- 클린 Windows PC에서 압축 해제 후 실행·저장·재실행

세부 일정과 완료 조건은 `09_MVP/wbs.md`를 따른다.
