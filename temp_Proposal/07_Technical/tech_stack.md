# 기술 구조와 협업 잠금

> **상태: LOCKED · 2026-07-19 · 기준 문서: `new/ICEBREAKER_SPEC_v2.md` v2.6**  
> 같은 상태를 여러 곳에서 계산하지 않는 것을 최우선으로 한다. 새 프레임워크나 범용 아키텍처는 만들지 않는다.

## 1. 기술 스택

- Unity 6 + C#
- 창 제어: UniWindowController (480×56 축소 ↔ 720×480 확장)
- 데이터: 로컬 JSON
- 버전 관리: GitHub 기능 브랜치 · 매일 통합
- P0 플랫폼: Windows
- 언어: 한국어 (영어는 일정 여유 시)

새 패키지, DI 프레임워크, 온라인 백엔드, 분석 SDK는 P0에 추가하지 않는다.

## 2. 역할과 소유권

| 담당 | 책임 |

### 단일 소유 규칙

- **게임 수치 변경은 민준, 파괴 판정은 시연, 전달받은 값의 화면 표시는 정환**이 맡아 같은 상태를 여러 곳에서 계산하지 않는다.
- 시연은 파괴 결과만 전달하고 자금·목적지 진행·구매·저장 상태를 직접 변경하지 않는다.
- 민준만 자금·목적지 진행·정비 구매·저장 상태를 변경한다.
- 정환은 전달받은 상태를 표시하지만 게임 수치와 정산 합계를 계산하지 않는다.
- 각 담당자는 가짜 데이터로 자기 기능을 먼저 확인할 수 있어야 한다.

## 3. 공용 데이터

- `IceDefinition`: 단계, HP, 기본 자금, 생성 조건
- `SpecialIceDefinition`: 확률, HP 배수, 자금 배수, 효과
- `MaintenanceDefinition`: ID, 계통, 단계, 효과, 비용, 선행 조건
- `MaintenanceNodeViewData`: ID, 계통, 현재·최대 단계, 상태, 현재·다음 효과 문구, 다음 비용, 선행 노드 ID
- `DestinationDefinition`: 목적지, 목표, 화물, 표시 순서
- `GameState`: 현재 상태, 남은 시간, 자금, 목적지 진행, 구매 상태, 첫 파괴 여부, 쇄빙 시작 가능 여부
- `SettlementSummary`: 작업 자금, 파괴 수, 실제 목적지 진행 증가, 목적지 도달 여부와 ID

## 4. 파괴 이벤트 계약

```text
IceDestroyed(
  iceInstanceId,
  tier,
  specialType,
  screenPosition,
  destroySource,
  destroyedAt
)
```

`destroySource`는 `Direct`, `Support`, `Chain` 중 하나다. 이 이벤트를 받은 진행 시스템(민준)이 다음을 한 번에 처리한다.

1. `iceInstanceId`가 이미 처리됐는지 확인한다.
2. 단계와 특수빙에 따른 자금을 계산한다.
3. 보유 정비 자금을 즉시 증가시킨다.
4. 현재 목적지 진행도가 목표 미만이면 진행도를 1 증가시킨다.
5. UI에 파괴 위치와 지급액을 전달한다.
6. 저장 상태를 변경됨으로 표시한다.

`StageEnded`에는 자금 증가값을 넣지 않는다.

### 작업별 누적과 정산

작업별 요약을 위해 `stageEarnedFunds`, `stageDestroyedCount`, `stageDestinationProgressGain`을 별도로 누적한다. 세 값은 자금 정산 화면에서 읽기만 하고 다음 쇄빙 작업 시작 시 0으로 초기화한다.

```text
SettlementSummary
- earnedFunds: long
- destroyedCount: int
- destinationProgressGain: int
- reachedDestination: bool
- destinationId: string 또는 null
```

- 진행 시스템은 타이머가 끝나는 순간 누적값을 읽기 전용 `SettlementSummary`로 고정한다.
- `SettlementReady(summary)`를 발생시키며 UI는 전달받은 값을 표시만 한다.
- `destroyedCount`는 직접·보조·연쇄를 포함해 승인된 고유 `IceDestroyed` 이벤트 수다.
- `destinationProgressGain`은 목표 초과분을 제외한 실제 목적지 진행 증가량이다. 목표 달성 뒤에는 `0`이 될 수 있다.
- 정산 UI는 이 데이터를 근거로 자금·목적지 진행·저장 상태를 다시 변경하지 않는다.

## 6. 시간 경계와 저장

- 얼음 HP가 0이 된 시점이 타이머 0 이전일 때만 파괴 이벤트를 인정한다.
- 타이머 0 이후 도착한 투사체와 연쇄 피해는 취소한다.
- 자금과 목적지 진행은 메모리에서 즉시 변경한다.
- 저장 파일 쓰기는 최대 1초 동안 묶어서 처리한다.
- 작업 종료, 정비 구매, 앱 정상 종료 때는 대기 중인 저장을 즉시 완료한다.
- 다음 쇄빙 가능 시각은 절대 시각으로 저장한다. 앱 종료 중에도 시간은 진행되지만 자금과 목적지 진행은 변하지 않는다.
- 설정 화면이 열려 있는 동안 게임을 일시정지한다. 쇄빙·항해 타이머, 얼음 생성, 진행 중 피해, 보조 파쇄 충전을 멈추고 닫으면 멈춘 시점부터 재개한다.

### 저장 필드

- 보유 정비 자금과 구매한 정비 항목
- 현재 목적지와 진행도, 완료한 목적지
- 첫 파괴 여부
- 다음 쇄빙 가능 시각 (절대 시각)
- 볼륨·화면 흔들림 등 설정

저장 후 재실행하면 자금·목적지 진행·구매·첫 파괴 여부가 유지돼야 한다.

## 7. 데이터 파일

- 표준 데이터: 항해 30초 · 목적지 목표 120/600/2,400 · 정비 비용 100%
- 시연 데이터: 항해 10초 · 목적지 목표 40/120/300 · 정비 비용 10% 올림
- 표준과 시연은 같은 코드와 규칙을 사용하고 데이터 파일만 분리한다. 저장 파일도 서로 공유하지 않는다.
- 공개 설정 화면에는 데이터 전환을 노출하지 않는다.

## 8. 창과 성능

- 480×56 축소 게임 바는 작업 표시줄 바로 위에 고정한다.
- 자동 확장·포커스 스틸·OS 알림 없음. 창 확장·축소 애니메이션 300ms 이내.
- 쇄빙 작업 중 최대 20개 얼음과 연쇄 동시 발생 기준으로 프레임을 확인한다.
- 얼음·파편·보상 숫자는 오브젝트 풀을 사용해 매 프레임 할당을 피한다.

## 9. 통합 원칙

- 매일 한 번 통합하고, 각 담당자는 가짜 데이터로 자기 기능을 먼저 확인한다.
- 실행 진입 연결과 최종 실행 파일은 민준이 담당한다. 별도 타이틀·메인 화면 없이 실행 시 바로 축소 런처를 띄운다.
- 화요일(7/21) 통합 뒤에는 새로운 핵심 시스템을 추가하지 않는다.
- 목·금(7/23~24)에는 진행을 막는 오류, 가독성, 밸런스, 아트·오디오, 제출 안정성을 우선한다.
- `.meta` 파일을 수동 삭제·이름 변경하지 않는다.

## 10. 검증

`new/ICEBREAKER_SPEC_v2.md` 12장의 시스템 시나리오 10개와 UI 시나리오 21개를 기준으로 한다. 핵심은 다음과 같다.

- T1 직접 파괴 → 자금 +10, 목적지 진행 +1, 보상 표시 1회
- 같은 얼음에 클릭·연쇄가 동시에 들어가도 지급은 한 번
- 타이머 0 이후 피해에는 자금·진행 없음
- 작업 종료 시 자금이 추가로 증가하지 않고, 정산 화면 획득액은 실제 지급 합계와 같음
- 저장 후 재실행하면 자금·목적지 진행·구매·첫 파괴 여부 유지
- 다른 PC에서 처음부터 목적지 도달까지 플레이 가능한 Windows 빌드
