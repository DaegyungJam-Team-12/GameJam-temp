# ICEBREAKER — P0 구현 잠금

> **상태: LOCKED · 2026-07-19 · 기준 문서: `new/ICEBREAKER_SPEC_v2.md` v2.6**
> 이 문서는 v2.6의 게임 규칙을 바꾸지 않고, 구현자가 다르게 해석할 수 있는 창·전투·이벤트·저장 경계를 고정한다.

## 0. 문서 권한과 범위

- 세계관, P0 범위, 밸런스 값은 통합 기획서와 `02_System/`, `03_Data/`를 따른다.
- 창 동작, 피해 계산 순서, 연쇄 깊이, 이벤트 필드, 저장·재실행 상태가 충돌하면 이 문서를 구현 기준으로 사용한다.
- 새 프레임워크, 범용 이벤트 버스, 진행 중 쇄빙 복원, 투명 창, 클릭 통과는 P0에 추가하지 않는다.
- Unity 에디터 정확한 버전과 프로젝트 템플릿은 프로젝트 생성 후 `ProjectSettings/ProjectVersion.txt`와 `Packages/packages-lock.json`을 단일 출처로 사용한다.

## 1. 창 동작 잠금

### 1.1 P0 지원 조건

| 항목 | 확정 규칙 |
|---|---|
| 플랫폼 | Windows 10·11 x64 |
| 그래픽 API | Direct3D 11 |
| 창 형태 | 불투명 직사각형 borderless 창 |
| 투명·클릭 통과 | P0 제외 |
| 공식 화면 환경 | 주 모니터 1대 · 아래쪽 taskbar · 배율 100%·125% |
| 기타 모니터·taskbar 배치 | best effort. 진행 불가 오류만 수정 |

- UniWindowController는 `Packages/manifest.json`과 `Packages/packages-lock.json`에 해석된 revision을 커밋해 팀 전원이 같은 소스를 사용한다.
- 창 플러그인이 초기화되지 않아도 게임은 일반 960×540 창으로 실행한다. 플러그인 실패로 앱을 종료하지 않는다.

### 1.2 크기와 위치

- 축소 창의 클라이언트 영역은 800×72px, 확장 창은 960×540px다.
- 두 크기는 DPI 배율을 곱하지 않는 클라이언트 픽셀 기준이다. Canvas는 같은 기준 해상도를 사용한다.
- P0 Canvas 기준 해상도는 축소 View 800×72, 확장 View 960×540이며 `Scale With Screen Size`와 앵커를 사용한다.
- P0에서는 사용자가 크기를 조절하지 않는다. 1000×90·1200×675와 1200×108·1440×810는 게임잼 이후 설정 후보다.
- 축소 창은 Windows 작업 영역의 오른쪽 아래에 놓는다. 작업 영역은 taskbar를 제외한 OS 제공 영역이다.
- 축소 창의 오른쪽과 아래쪽은 작업 영역 경계에서 각각 8px 떨어뜨린다. 아래쪽 taskbar 환경에서는 taskbar 바로 위에 표시된다.
- 확장할 때 축소 창의 아래쪽 기준점을 유지하고 위쪽과 왼쪽으로 크기를 늘린다.
- 확장 창이 작업 영역을 벗어나면 전체 창이 보이도록 작업 영역 안으로 보정한다.
- 사용자가 창을 임의로 드래그하거나 크기를 조절하는 기능은 P0에 두지 않는다.

### 1.3 포커스와 항상 위

- 앱이 보이는 동안 축소·관리·쇄빙 창은 항상 위 상태를 유지한다.
- 항해 타이머가 끝나도 창 크기, 위치, 포커스를 바꾸지 않는다. 시작 버튼만 한 번 점등한다.
- 사용자가 정비, 운항 현황, 쇄빙 시작, 설정을 직접 눌렀을 때만 포커스를 받을 수 있다.
- 쇄빙 중 다른 앱으로 포커스가 이동해도 60초 타이머는 계속 진행한다.
- 설정 모달을 연 경우에만 게임 시간을 일시정지한다.

### 1.4 상태별 창 전환

| 상태 | 창 | 포커스·입력 |
|---|---|---|
| 항해 중 | 800×72 | 자동 포커스 없음 |
| 쇄빙 준비 완료 | 800×72 | 자동 포커스 없음 · 시작 버튼만 활성 |
| 정비·운항 현황 | 960×540 | 사용자의 버튼 입력으로 확장 |
| 카운트다운·쇄빙 | 960×540 | 사용자의 시작 입력으로 확장 |
| 작업 종료·정산·도착 | 960×540 | 정산 규칙에 따라 입력 잠금 |
| 운항 완료 | 800×72 | `운항 완료` 표시 · 쇄빙 시작 비활성 |

- P0에서는 크기를 즉시 전환한다. 별도 확장·축소 애니메이션은 구현하지 않는다. 즉시 전환도 300ms 이내 조건을 만족한다.
- 북쪽 기지 이후 자유 쇄빙은 허용 예정이지만 게임잼 빌드에는 포함하지 않는다.

## 2. 전투·강화 계산 잠금

### 2.1 자료형과 계산 순서

- 얼음 HP와 모든 피해는 `float`로 계산한다.
- 피해마다 반올림하지 않는다. 피해 적용 뒤 HP가 0 이하이면 파괴한다.
- 정비 자금만 모든 배수를 적용한 뒤 한 번 내림해 `long`으로 저장한다.
- `현재 클릭 피해`는 치명타를 적용하지 않은 D01 반영 피해다.

```text
현재 클릭 피해 = 기본 클릭 피해 × 1.6^(D01 단계)
직접 치명타 피해 = 현재 클릭 피해 × 3
D03 전이 피해 = 실제 초과 피해 × 0.5
H01 피해 = 현재 클릭 피해 × (H01 단계 × 0.25)
균열빙 피해 = 현재 클릭 피해 × 3 × H02 효과 배수
H03 피해 = 현재 클릭 피해 × 1.5
```

- 치명타는 직접 클릭·홀드 공격에만 5% 확률로 발생한다.
- D03, 보조탄, H01, 결정빙, 균열빙, H03 피해에는 자체 치명타가 없다.
- 치명타 직접 공격으로 발생한 D03 초과 피해는 실제 치명타 피해의 초과분을 기준으로 계산하되 다시 치명타 판정하지 않는다.

### 2.2 직접 입력과 S01 충전

- 직접 공격 입력은 마우스 왼쪽 버튼만 사용한다.
- 버튼을 누르는 순간 커서 아래 얼음에 즉시 한 번 공격한다.
- 누른 상태에서는 `5 + D02 단계 × 2`회의 초당 빈도로 다시 공격한다. 최대 초당 11회다.
- 각 홀드 틱은 그 시점의 커서 아래 얼음을 다시 판정한다. 이전 대상이 파괴돼도 자동으로 다른 얼음을 추적하지 않는다.
- 살아 있는 얼음에 직접 피해가 실제 적용됐을 때만 `유효한 직접 입력` 1회로 센다.
- 빈 공간, 입력 잠금 상태, 이미 파괴된 얼음, 타이머 0 이후 입력은 S01을 충전하지 않는다.
- S01 충전은 같은 쇄빙 작업 안에서 대상이 바뀌거나 설정으로 일시정지돼도 유지한다.
- 쇄빙 작업을 시작할 때 0으로 초기화하고 작업 종료·중단 시 폐기한다. 저장하지 않는다.
- 12번째 유효 입력의 직접 피해 처리가 끝난 뒤 보조탄을 발사하고 충전을 0으로 되돌린다.

### 2.3 보조 파쇄 표적

- S01 기본 표적은 보조 파쇄 장비의 발사 위치에서 가장 가까운 유효 얼음이다.
- S02 단계마다 서로 다른 추가 표적 1개를 선택한다. 기본 표적 피해는 현재 클릭 피해 100%, 추가 표적은 각각 70%다.
- S03 구매 후에는 특수빙을 우선한다. 특수빙이 없으면 남은 HP가 가장 높은 얼음을 우선한다.
- S03이 특수빙을 공격하면 해당 보조탄 피해에 ×2를 적용한다.
- 같은 발사에서 한 얼음을 두 번 지정하지 않는다.

### 2.4 공통 표적 동률 처리

유효 표적은 살아 있고 화면에 활성화돼 있으며, 해당 효과의 재생성 보호를 받지 않는 얼음이다. 동률은 항상 다음 순서로 결정한다.

```text
효과별 우선 조건
→ 남은 HP가 높은 대상
→ 기준점에서 가까운 대상
→ iceInstanceId가 작은 대상
```

- D03은 파괴 위치에서 가장 가까운 다른 얼음을 선택한다. 대상이 없으면 초과 피해를 버린다.
- 기본 보조탄은 장비 발사 위치에서 가까운 순서, S03은 특수빙·HP·거리·ID 순서다.
- 결정빙 파편은 자신보다 낮은 단계의 서로 다른 얼음을 파괴 위치에서 가까운 순서로 선택한다.
- 결정빙 파편 수보다 유효 표적이 적으면 남은 파편은 소멸한다.

### 2.5 연쇄 컨텍스트와 깊이

- 유효한 직접 입력마다 새 `chainId`를 발급한다.
- 같은 입력에서 발생한 직접 공격과 S01 보조탄은 같은 `chainId`를 사용한다.
- 직접 공격과 보조탄의 최초 피해·파괴는 `chainDepth 0`이다.
- 파괴로 발생한 D03, 특수빙, H01 효과는 원인 파괴보다 깊이를 1 증가시킨다.
- 최대 깊이는 3이다. depth 3 피해로 얼음은 파괴될 수 있지만 그 파괴에서 D03·특수빙·H01을 다시 만들지 않는다.
- 같은 얼음에 여러 피해가 예약돼도 순서대로 처리하고, 처음 HP가 0 이하가 된 순간 한 번만 파괴한다. 이후 예약 피해는 취소한다.
- 연쇄 처리는 깊이 우선이 아니라 같은 depth를 먼저 처리하는 너비 우선 큐를 사용한다.
- 한 파괴에서 여러 효과가 생기면 `D03 → 특수빙 → H01` 순서로 큐에 넣는다.

### 2.6 H03 5연쇄

- 같은 `chainId`에서 승인된 파괴 수를 센다. depth 0의 최초 직접·보조 파괴도 포함한다.
- 다섯 번째 파괴가 승인되는 순간 그 위치에 H03을 한 번 예약한다.
- H03은 한 `chainId`에서 최대 한 번 발동한다.
- 다섯 번째 파괴가 depth 3이어도 H03 피해는 발동한다. 이 경우 H03 피해와 그 파괴 결과를 depth 3으로 기록하고 추가 연쇄 효과는 만들지 않는다.

### 2.7 H02 특수빙 증폭

| H02 단계 | 결정빙 파편 수 | 균열빙 피해 배수 | 균열빙 반경 |
|---:|---:|---:|---:|
| 0 | 5 | ×1.0 | 120px |
| 1 | 7 | ×1.3 | 156px |
| 2 | 8 | ×1.6 | 192px |

- 결정빙의 자금 배수 ×4는 H02로 증가하지 않는다.
- 결정빙 파편은 HP 피해가 아니라 유효한 낮은 단계 얼음을 즉시 파괴한다.
- 화면 문구는 `특수빙 효과 규모 +30%`로 표시한다.

### 2.8 재생성 보호

- 새로 생성된 얼음은 0.25초 동안 Direct 이외의 피해와 표적 선택에서 제외한다.
- 클릭·홀드 직접 공격은 재생성 보호 중에도 적용할 수 있다.
- 보호 시간은 설정 일시정지 동안 진행하지 않는다.

## 3. 공용 이벤트 계약

### 3.1 ID와 좌표

- `stageId`: 쇄빙 작업을 시작할 때 진행 시스템이 발급하는 증가형 `long`.
- `iceInstanceId`: 얼음을 생성할 때 파쇄 시스템이 발급하는 증가형 `long`. 오브젝트 풀 인스턴스 ID를 재사용하지 않는다.
- `chainId`: 유효한 직접 입력마다 파쇄 시스템이 발급하는 증가형 `long`.
- 세 ID의 증가 범위는 현재 앱 실행 세션이다. 재실행 시 진행 중 작업과 이벤트를 복원하지 않으므로 카운터를 다시 시작해도 된다.
- `referencePosition`: 960×540 기준 해상도의 좌하단 원점 `Vector2`. UI가 자신의 Canvas 좌표로 변환한다.
- `stageElapsedSeconds`: 카운트다운 종료를 0으로 하는 읽기 전용 작업 시계의 `double`. 설정 일시정지 시간은 포함하지 않는다.
- 모든 UTC 문자열은 ISO 8601 round-trip 형식(`O`)으로 직렬화한다.

### 3.2 효과 분류

```text
DestroyCategory = Direct | Support | Chain
EffectType = Click | Hold | Overkill | SupportShot |
             CrystalShard | CrackExplosion | HullFragment | IceCollapse
```

- `DestroyCategory`는 정산·통계용 큰 분류다.
- `EffectType`은 VFX·SFX와 재현 로그용 실제 원인이다.

### 3.4 발행과 소유권

| 이벤트 | 발행 | 소비 |
|---|---|---|
| `StageStarted` | 진행 시스템 | 파쇄·UI·저장 |
| `DamageAppliedEvent` | 파쇄 시스템 | UI·오디오 |
| `SupportChargeChangedEvent` | 파쇄 시스템 | UI |
| `IceDestroyedEvent` | 파쇄 시스템 | 진행 시스템·UI·오디오 |
| `RewardGrantedEvent` | 진행 시스템 | UI·정산 누적·저장 |
| `StageEnded` | 진행 시스템 | 파쇄·UI·저장 |
| `SettlementReady` | 진행 시스템 | 정산 UI |

- 파쇄 시스템은 자금, 목적지, 구매, 저장 데이터를 직접 변경하지 않는다.
- 진행 시스템은 승인된 `IceDestroyedEvent`의 `stageId`와 `iceInstanceId` 중복을 확인한 뒤 자금·목적지를 한 번만 변경한다.
- UI는 `RewardGrantedEvent`의 금액과 위치를 표시하며 보상 값을 다시 계산하지 않는다.
- 정산 누적은 승인된 `RewardGrantedEvent`와 대응하는 고유 파괴만 사용한다.
- 이벤트 전달은 명시적 C# event 또는 인터페이스로 구현한다. 범용 이벤트 버스와 리플렉션 기반 메시지 시스템을 추가하지 않는다.

### 3.5 타이머 0 경계

- 진행 시스템의 작업 시계만 쇄빙 입력 허용 여부를 결정한다.
- 파쇄 시스템은 피해를 적용하기 직전에 `Playing` 상태와 `stageElapsedSeconds < 60`을 확인한다.
- 시계가 60초에 도달하면 상태를 먼저 `StageEnding`으로 변경하고 입력, 예약 피해, 투사체, 연쇄 큐, 재생성을 취소한다.
- 상태 변경 전에 HP가 0 이하가 되어 발행된 이벤트만 승인한다.
- `StageEnded`에는 자금이나 파괴 결과를 넣지 않는다.
- 상태 변경 전에 동기적으로 발행돼 승인 대기 중인 이벤트만 처리한 뒤 누적값을 읽기 전용 `SettlementSummary`로 고정하고 `SettlementReady`를 발생시킨다.

## 4. 저장·종료·재실행 잠금

### 4.1 저장 필드

```text
saveVersion: int
profileId: string
funds: long
maintenanceLevels: List<{ id: string, level: int }>
currentDestinationIndex: int
destinationProgress: int
completedDestinationIds: List<string>
pendingArrivalDestinationId: string 또는 null
firstDestroyShown: bool
nextAvailableAtUtc: string 또는 null
runInProgress: bool
gameCompleted: bool
masterVolume: float
screenShakeEnabled: bool
```

- `SettlementSummary`, S01 충전, 활성 얼음, 남은 HP, 연쇄 큐, 남은 쇄빙 시간은 저장하지 않는다.
- 표준 빌드는 `profileId = standard`와 `save_standard.json`, 시연 빌드는 `profileId = demo`와 `save_demo.json`을 사용한다.
- 표준·시연은 같은 코드를 사용하고 빌드별 읽기 전용 `profileId`와 데이터 파일만 다르게 만든다. 공개 설정에서 전환하지 않는다.

### 4.2 저장 시점과 파일 쓰기

- 파괴 보상은 메모리에서 즉시 반영하고 최대 1초 단위로 묶어 저장한다.
- 쇄빙 시작, 작업 종료, 정비 구매, 목적지 도달 예약, 목적지 전환, 설정 변경, 정상 종료 때 대기 중인 저장을 즉시 완료한다.
- 쇄빙 시작 시 `runInProgress = true`로 저장한다.
- 정산과 필요한 도착 처리가 끝나 축소 화면으로 돌아갈 때 `runInProgress = false`로 저장한다.
- JSON은 임시 파일에 먼저 기록하고 쓰기가 성공하면 기존 저장 파일을 교체한다.
- 저장 파싱에 실패하면 손상 파일을 덮어쓰지 않고 별도 이름으로 보존한 뒤 새 저장으로 시작한다.

### 4.3 최초 실행과 항해 시각

- 저장 파일이 없는 최초 실행만 즉시 `쇄빙 준비 완료`로 시작한다.
- 정상 반복에서는 축소 화면으로 돌아가는 시각에 `nextAvailableAtUtc = 현재 UTC + 항해 시간`을 저장한다.
- 시스템 시각이 앞으로 이동해 목표 시각을 지났으면 즉시 준비 완료다.
- 시스템 시각이 뒤로 이동하면 남은 시간을 현재 프로필의 전체 항해 시간보다 크게 표시하지 않는다.
- 항해 중 설정을 열면 일시정지 시작 시각을 기록하고, 정상적으로 닫을 때 열린 시간만큼 `nextAvailableAtUtc`를 뒤로 이동한 뒤 저장한다.
- 카운트다운·쇄빙 중에는 작업 시계만 멈춘다. `nextAvailableAtUtc`는 변경하지 않는다.
- 준비 완료 상태에는 진행 중인 타이머가 없으므로 설정을 닫아도 시각을 변경하지 않는다.
- 설정 중 강제 종료된 경우 설정 화면을 복원하지 않는다. 항해 중이었다면 마지막 저장의 절대 시각을 사용하고, 쇄빙 중이었다면 진행 중 종료 규칙을 따른다.

### 4.4 진행 중 종료 처리

진행 중인 카운트다운·쇄빙·작업 종료·정산·도착 화면은 재실행 시 복원하지 않는다.

| 종료 시점 | 유지 | 재실행 상태 |
|---|---|---|
| 항해 중 | 모든 저장 상태·절대 시각 | 남은 항해 또는 준비 완료 |
| 준비 완료 | 모든 저장 상태 | 준비 완료 |
| 카운트다운·쇄빙 중 | 마지막 저장 완료 시점까지 승인된 자금·목적지·구매 | 전체 항해 시간으로 다시 시작 |
| 작업 종료·정산 중 | 이번 작업에서 저장 완료된 결과 | 전체 항해 시간으로 다시 시작 |
| 목적지 도착 처리 중 | 승인 결과·pending arrival | 목적지 전환을 한 번 완료한 뒤 전체 항해 시작 |
| 설정 중 | 마지막 완료 저장 | 설정을 닫은 상태로 복귀 |

- 부팅 시 `runInProgress = true`이면 진행 중 작업을 중단한 것으로 처리한다.
- 저장 완료된 보상을 회수하거나 같은 작업을 복원하지 않는다. 비정상 종료에서는 저장 지연 때문에 최대 1초 이내의 마지막 보상이 유실될 수 있다.
- 필요한 목적지 전환을 먼저 완료한 뒤 `runInProgress = false`, `nextAvailableAtUtc = 현재 UTC + 항해 시간`으로 저장한다.
- `gameCompleted = true`이면 다른 재실행 분기보다 완료 상태를 우선한다.

### 4.5 목적지 도착의 원자성

- 목적지 진행이 목표에 처음 도달한 순간 `pendingArrivalDestinationId`를 저장한다. 이후 파괴는 자금만 지급한다.
- 정상 흐름에서는 작업 종료 정산에 도착 배지를 표시하고, 정산 뒤 공통 도착 연출을 재생한다.
- 도착 연출이 끝나면 다음 순서로 한 번에 처리하고 즉시 저장한다.

```text
완료 목적지 목록에 추가
→ 다음 목적지로 이동
→ 목적지 진행 0
→ pendingArrivalDestinationId 제거
```

- 부팅 시 pending arrival이 있으면 완료 목록과 현재 목적지를 확인해 아직 적용되지 않은 전환만 수행한다.
- 같은 목적지를 두 번 완료 목록에 추가하거나 진행도를 두 번 초기화하지 않는다.
- 비정상 종료 복구에서는 정산과 도착 연출을 다시 재생하지 않는다.

### 4.6 북쪽 기지와 게임잼 완료 상태

- 북쪽 기지 도착 처리가 끝나면 `gameCompleted = true`로 저장한다.
- 현재 목적지는 북쪽 기지, 진행은 목표값으로 유지하고 `pendingArrivalDestinationId`를 제거한다.
- 축소 화면에는 `운항 완료`를 표시하고 쇄빙 시작 버튼을 비활성화한다.
- 정비·운항 현황·설정은 계속 열 수 있다.
- 북쪽 기지 이후 자금만 얻는 자유 쇄빙은 게임잼 이후 기능으로 두며 P0에서는 시작할 수 없다.

## 5. 통합 검증 시나리오

1. **직접 파괴:** 치명타를 비활성화한 검증 설정에서 D01 0단계 T1을 일반 직접 공격 10회로 파괴하면 자금 +10, 목적지 +1, 파괴·보상 이벤트가 각각 한 번 발생한다.
2. **치명타:** 직접 치명타만 현재 클릭 피해 ×3이며, 그 공격이 만든 D03·H01에는 별도 치명타가 발생하지 않는다.
3. **S01 충전:** 살아 있는 얼음에 유효한 직접 피해 12회가 적용된 뒤 같은 입력의 직접 처리가 끝나면 보조탄을 한 번 발사하고 충전이 0이 된다.
4. **H02 결정빙:** H02 0·1·2단계의 결정빙은 서로 다른 낮은 단계 얼음을 최대 5·7·8개 즉시 파괴한다.
5. **H02 균열빙:** H02 0·1·2단계에서 균열빙 반경은 120·156·192px이고 피해 배수는 ×1.0·×1.3·×1.6이다.
6. **최대 깊이:** depth 3 피해로 얼음이 파괴돼도 D03·특수빙·H01이 새로 예약되지 않는다.
7. **H03:** 같은 chainId의 다섯 번째 승인 파괴 위치에서 한 번만 발동하며 최초 depth 0 파괴도 수에 포함한다.
8. **중복 파괴:** 같은 iceInstanceId에 직접·보조·연쇄 피해가 겹쳐도 `IceDestroyedEvent`와 보상은 한 번만 승인된다.
9. **타이머 0:** 60초 도달 뒤 남은 투사체와 연쇄 큐는 피해·자금·목적지를 만들지 않는다.
10. **쇄빙 중 종료:** 저장 완료된 보상은 유지하고 활성 얼음·남은 시간은 복원하지 않으며 재실행 시 전체 항해 시간을 시작한다.
11. **목적지 도착 중 종료:** pending arrival을 한 번만 적용하고 중복 완료·중복 진행 초기화 없이 항해로 복귀한다.
12. **북쪽 기지:** 도착 처리 뒤 `운항 완료`를 표시하고 게임잼 빌드의 쇄빙 시작을 비활성화한다.

## 6. 구현 완료 체크

- 창 위치 계산은 OS 작업 영역을 사용하고 화면 전체 높이를 직접 빼지 않는다.
- 파쇄 시스템은 자금·목적지·저장 상태를 변경하지 않는다.
- UI는 피해·보상·정산 수치를 다시 계산하지 않는다.
- 모든 파괴와 보상 로그에 stageId, iceInstanceId, chainId가 남는다.
- 진행 중 작업을 저장·복원하는 코드를 만들지 않는다.
- 표준·시연 저장 파일이 서로 섞이지 않는다.
- 북쪽 기지 이후 자유 쇄빙을 게임잼 P0에 추가하지 않는다.
