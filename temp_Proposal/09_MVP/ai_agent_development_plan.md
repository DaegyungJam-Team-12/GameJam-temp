# ICEBREAKER — AI 에이전트 협업 개발 계획

> **상태: ACTIVE · 2026-07-19 · 기준: 통합 기획서 v2.6**
> 목표: 2026-07-21 화요일 밤까지 P0 기능을 연결하고, 7월 22일 수요일에는 새 기능 없이 Windows 빌드·회귀 검증·진행 불가 오류 수정만 한다.

## 1. 전제와 결론

세 개발자 모두 Codex·Claude Code 계열 AI 에이전트를 사용한다. 따라서 사람별로 큰 기능 하나를 며칠간 맡기는 방식보다, **1~4시간 안에 검토 가능한 Task → 전용 브랜치 → PR → 검증 후 `dev` 병합**을 반복한다.

AI 에이전트가 초안을 빠르게 만들 수 있어도 다음 책임은 사람에게 남는다.

- 기획 해석과 범위 결정
- 변경 파일과 diff 검토
- Unity 컴파일·Play Mode·Windows 실행 검증
- Scene·Prefab·직렬화 참조 확인
- PR 승인과 병합

예상 소요시간은 에이전트가 코드를 생성하는 시간만이 아니다. **Task 전달, 에이전트 실행, 사람 검토, 수정 1회, Unity 검증, PR 작성**을 포함한 경과시간이다.

## 2. 역할 배치

| 역할 | 권장 대상 | 책임 | 업무 비중 |
|---|---|---|---:|
| `LEAD` — 기술 리드·통합 | 민준 | 프로젝트 기반, 공용 계약, 상태 흐름, 진행·경제, 정비, 저장, 창 제어, Main Scene, 통합·빌드 | 약 45% |
| `GAMEPLAY` — 파쇄 시스템 | Unity 게임 1회 + Git·OOP 경험 개발자 | 입력, 얼음, 피해·파괴, 재생성, 특수빙, 보조 파쇄, 연쇄, 성능·예외 | 약 28% |
| `UIQA` — View·피드백·QA | Unity·Git 초급 개발자 | UI Sandbox, View Prefab, HUD, 보상·정산 표시, 기본 피드백, 정형 QA | 약 27% |
| `ART` — 아트 제작 | 2D 아티스트 | 콘셉트 마스터, 게임 오브젝트, UI 그래픽, 스프라이트 시트, Google Drive 납품 | 개발 비중 외 병렬 작업 |
| `PLAN` — 플레이테스트·밸런스 | 기획자 | Unity 플레이테스트, GitHub Issue, 밸런스 수치 제안, 문구·제출 QA | 개발 비중 외 병렬 작업 |

업무 비중 퍼센트는 세 개발자의 개발 작업만 합산한 값이다.

### 역할 경계

- `LEAD`만 자금, 목적지 진행, 정비 구매, 저장 상태를 변경한다.
- `GAMEPLAY`는 파괴를 판정하고 `IceDestroyedEvent`를 발행하지만 자금·목적지·저장을 변경하지 않는다.
- `UIQA`는 전달받은 `GameState`, `RewardGrantedEvent`, `SettlementSummary`를 표시하지만 수치를 다시 계산하지 않는다.
- `Main.unity`, 공용 계약, 패키지·프로젝트 설정은 `LEAD`만 최종 수정한다.
- `GAMEPLAY`와 `UIQA`는 각자의 Sandbox Scene과 Prefab을 사용하고 Main Scene 연결은 `LEAD`가 한다.
- `ART`는 콘셉트 표현에 자율권을 갖되 해상도·정보 우선순위·필수 상태·납품 규격은 고정한다.
- `PLAN`은 밸런스 값을 직접 수정하지 않고 플레이 기록과 변경 수치를 Issue로 제안한다.

## 3. 모듈·파일 소유권

Unity 프로젝트 생성 후 아래 경계를 유지한다. 실제 폴더명이 달라져도 소유 원칙은 바꾸지 않는다.

| 영역 | 소유자 | 다른 담당자의 변경 방식 |
|---|---|---|
| `Assets/ICEBREAKER/Shared/` | `LEAD` | 계약 변경 요청을 PR 댓글로 남기고 `LEAD`가 반영 |
| `Assets/ICEBREAKER/Core/`, `Progression/`, `Save/`, `Window/` | `LEAD` | 직접 수정 금지 |
| `Assets/ICEBREAKER/Gameplay/` | `GAMEPLAY` | 통합 수정은 소유자와 합의 |
| `Assets/ICEBREAKER/UI/`, `VFX/`, `Audio/` | `UIQA` | 상태 바인딩 코드는 `LEAD`가 별도 파일로 연결 가능 |
| `QA/Reports/` | `UIQA` | 테스트 결과·캡처 목록·재현 절차를 Task ID별로 기록 |
| `Assets/ICEBREAKER/Scenes/Main.unity` | `LEAD` | 다른 담당자는 수정 금지 |
| `GameplaySandbox.unity` | `GAMEPLAY` | 게임플레이 담당 전용 |
| `UISandbox.unity` | `UIQA` | UI·피드백 담당 전용 |
| `ProjectSettings/`, `Packages/` | `LEAD` | 사전 승인 없는 변경 금지 |
| Google Drive 아트 원본·PNG | `ART` | `UIQA`가 관련 에셋 통합 브랜치로 가져오고 `LEAD`가 Main Scene 연결 |
| GitHub Issue·Project | `PLAN`, `LEAD` | 기획 문서 PR 대신 버그·밸런스·릴리스 검증 결과를 기록 |

`.meta` 파일은 대응 에셋과 같은 PR에 반드시 포함한다. AI 에이전트가 관련 없는 Scene, Prefab, ProjectSettings, 기획 문서를 수정하면 해당 변경은 제거한다.

## 4. 의존성 구조

```text
FND-01 Unity 프로젝트·공용 계약
├─ CORE-00 상태·진행 뼈대 ───────────────┐
├─ GP-01 T1 파쇄 → GP-02~08 ────────────┼─ INT-01~03 ── REL-01~02
├─ UI-01 UI Sandbox → UI-02~06 ─────────┤
└─ WIN-00 창 플러그인 Spike → WIN-01 ──┘
```

- `FND-01`이 `dev`에 병합되기 전에는 기능 코딩을 시작하지 않는다.
- 후속 Task는 의존 Task가 `dev`에 병합된 커밋에서 브랜치를 만든다.
- 미병합 브랜치에서 다시 브랜치를 만드는 stacked PR은 `LEAD`가 승인한 경우만 허용한다.
- UI는 실제 게임플레이 완성을 기다리지 않고 `LEAD`가 제공한 가짜 `GameState`와 이벤트 샘플로 개발한다.
- 게임플레이는 실제 진행 시스템을 기다리지 않고 이벤트 로그와 자동 검증용 소비자로 확인한다.
- 정비·보조·연쇄도 `FND-01`의 `MaintenanceDefinition`·`CombatConfig` 계약과 가짜 값으로 병렬 개발하고 실제 구매 결과 연결은 `INT-03`에서 한다.

## 5. 브랜치와 PR 흐름

### 브랜치 구조

```text
main                         검증된 기준선·릴리스 후보
└─ dev                       일일 통합
   ├─ feature/<task-id>-<name>
   ├─ fix/<task-id>-<name>
   └─ test/<task-id>-<name>
```

1. 이 계획과 v2.6 기준선을 `main`에 병합한다.
2. 개발 시작 전에 `dev`를 최신 `main`과 동기화한다.
3. 모든 WBS Task는 최신 `dev`에서 전용 브랜치를 만든다.
4. 기능 PR은 `dev`를 대상으로 한다.
5. 수요일 검증을 통과한 `dev`만 `main`으로 릴리스 PR을 만든다.

### Task당 Git 절차

```text
git fetch origin
git switch dev
git pull --ff-only
git switch -c feature/<task-id>-<name>
작업·검증
git push -u origin feature/<task-id>-<name>
PR → dev
```

- 한 브랜치에는 WBS Task 하나만 넣는다.
- 한 사람은 동시에 최대 2개 PR만 연다.
- 강제 푸시, 공용 브랜치 직접 커밋, 관련 없는 정리·리팩터링을 금지한다.
- `UIQA` 담당은 rebase보다 최신 `dev` 병합 또는 새 브랜치 재생성을 사용한다.
- 통합은 최소 하루 2회, 화요일에는 Task 완료 즉시 수행한다.

아트와 기획 지원 Task에는 다음 예외를 적용한다.

- 아티스트는 Git 브랜치나 PR을 만들지 않는다. Drive 납품물을 `UIQA`가 관련 Unity 에셋 PR에 포함한다.
- 기획자는 문서 PR을 만들지 않는다. 플레이테스트·밸런스·문구 결과를 GitHub Issue와 Project 상태로 관리한다.
- `LEAD`는 아티스트·기획자에게 전달 PPT와 역할별 작업전달서를 먼저 공유하고, 15분 설명 뒤 첫 Task를 배정한다.

## 6. AI 에이전트 Task 전달 형식

각 Task를 에이전트에 넘길 때 다음 항목을 빠뜨리지 않는다.

```text
Task ID:
목표:
기준 브랜치:
선행 Task/커밋:
반드시 읽을 문서:
수정 허용 경로:
수정 금지 경로:
입력 계약:
출력 계약:
완료 조건:
검증 명령 또는 Unity 수동 절차:
비범위:
```

### 에이전트 공통 지시

- 관련 기획 문서와 `07_Technical/implementation_lock.md`를 먼저 읽는다.
- Task 완료 조건을 만족하는 최소 변경만 한다.
- 공용 계약이 부족하면 임의로 확장하지 않고 담당자에게 요청한다.
- 다른 담당자 소유 파일과 Main Scene을 수정하지 않는다.
- 새 패키지, 프레임워크, 범용 이벤트 버스, 싱글턴 구조를 임의로 추가하지 않는다.
- 실제 실행하지 않은 테스트를 통과했다고 쓰지 않는다.
- 변경 파일, 검증 결과, 남은 위험을 PR 설명에 기록한다.

## 7. PR 검토 등급

WBS 난이도는 코드량보다 통합·직렬화·상태 손상 위험을 기준으로 한다.

| 난이도 | 의미 | 필수 검토 |
|---:|---|---|
| 1 | 문구·정적 데이터·단순 배치 | 담당자 자체 diff 확인 |
| 2 | 독립 View·낮은 위험 로직 | 담당자 자체 테스트 + 동료 1명 빠른 확인 |
| 3 | 한 모듈 안의 일반 기능 | 동료 리뷰 + Unity 수동 테스트 |
| 4 | 모듈 간 계약·전투 예외·저장 연동 | `LEAD` 리뷰 + 재현 가능한 테스트 |
| 5 | 상태 흐름·저장 복구·창 플러그인·최종 통합 | `LEAD` 직접 검증 + Windows 실기 확인 |

`LEAD`가 작성한 난이도 4~5 PR은 `GAMEPLAY` 담당자가 역리뷰한다. 코드 작성자가 자신의 에이전트 결과를 혼자 승인하지 않는다.

## 8. PR 완료 조건

모든 PR은 다음을 만족해야 병합할 수 있다.

- WBS Task ID와 선행 PR을 설명에 기록
- 허용된 파일만 변경
- Unity 컴파일 오류 없음
- `.meta` 누락 없음
- 완료 조건을 직접 확인한 결과 기록
- Scene·Prefab 변경 시 캡처 또는 수동 확인 절차 첨부
- 시간·랜덤·저장 관련 기능은 경계 조건 1개 이상 확인
- 알려진 제한과 후속 Task 기록
- `git diff --check` 통과

AI 에이전트가 생성한 코드가 동작해도 사람이 diff를 설명할 수 없으면 병합하지 않는다.

## 10. AI 사용에 따른 주요 위험과 대응

### 공용 구조 중복 생성

서로 다른 에이전트가 GameManager, 이벤트 버스, 데이터 모델을 각각 만들 수 있다. 공용 계약과 상태 변경은 `LEAD` 소유 파일 하나만 사용한다.

### Scene·Prefab 충돌

에이전트는 YAML 충돌을 의미 있게 해결하지 못할 수 있다. 담당별 Sandbox Scene과 Prefab을 사용하고 Main Scene은 `LEAD`가 연결한다.

### 빠른 코드 생성 뒤 검증 누락

Task 예상시간의 최소 30%를 사람 검토와 Unity 실행에 사용한다. 에이전트 답변의 테스트 통과 문구보다 실제 Editor Console과 빌드를 우선한다.

### 초급 담당자의 과도한 범위 확장

`UIQA` Task에는 수정 허용 경로와 가짜 입력 데이터를 제공한다. 로직이 필요해지면 직접 구현하지 않고 `LEAD`에게 계약을 요청한다.

### Windows 전용 기능의 마지막 날 실패

`WIN-00`에서 첫날 플러그인 초기화와 fallback 빌드를 확인한다. Windows PC 담당자를 정해 매일 실행한다. Windows 실기 환경이 없으면 P0 일정의 차단 요소로 취급한다.

## 11. 일정 부족 시 컷 순서
